UML & Analisis Sistem

Pertemuan 6 — UML: Class Diagram

Tujuan

  • Memahami peran UML sebagai bahasa komunikasi.
  • Mengurai komponen & relasi Class Diagram.
  • Membaca & membuat Use Case, Sequence, dan Activity Diagram.

Output

  • Class Diagram domain pilihan (tugas utama).
  • Use Case Diagram (opsional).
  • Tugas Individu: Activity Diagram.

Konsep Asal

Materi ini disarikan dan dibangun dari hasil analisis pada tugas individu Anda mengenai sistem e‑commerce, streaming, media sosial, transportasi, dan akademik.

Agenda

  1. Gambaran Singkat UML: Apa itu UML dan mengapa penting?
  2. Use Case Diagram: Memetakan kebutuhan dan interaksi pengguna.
  3. Class Diagram: Fokus utama pada struktur statis & relasi data.
  4. Sequence & Activity Diagram: Memvisualisasikan alur interaksi & proses.
  5. Aktivitas Kelas: Merancang diagram dari studi kasus.
  6. Tugas & Penilaian: Rubrik dan instruksi tugas individu.

UML Secara Singkat

Mengapa UML?

  • Bahasa visual standar untuk merancang & mengkomunikasikan sistem.
  • Berfungsi sebagai **"blueprint"** untuk mengurangi miskomunikasi antar stakeholder.
  • Menjembatani tahap analisis (**apa**) ke tahap desain (**bagaimana**).

Empat Diagram Utama dari Tugas Anda

  • Use Case — Siapa pengguna dan apa tujuannya.
  • Class — Struktur data dan relasinya.
  • Sequence — Urutan interaksi antar objek.
  • Activity — Alur kerja atau proses.
Alur kerja yang umum: Mulai dari Use Case (ruang lingkup) → Class (struktur) → Sequence/Activity (perilaku).

Use Case Diagram

Fungsi & Kegunaan

Use Case Diagram menggambarkan hubungan interaksi antara sistem dan **aktor** (pengguna atau sistem lain).

Diagram ini mendeskripsikan tipe interaksi dan tujuan yang ingin dicapai pengguna dengan sistem, menjadikannya langkah awal yang baik untuk pemodelan.

Sangat efektif untuk mendefinisikan ruang lingkup fungsional sebuah sistem dari sudut pandang pengguna.

Class Diagram

Struktur Statis Sistem

Class Diagram adalah diagram utama yang digunakan untuk menunjukkan **struktur statis** dari sebuah sistem.

Ini mencakup:

  • Kelas-kelas yang ada dalam sistem.
  • Atribut (data) dan **operasi** (metode) dari setiap kelas.
  • Hubungan antar kelas tersebut (seperti asosiasi atau pewarisan).

Relasi Antar Kelas (Relationships)

Class Diagram tidak hanya tentang kelas, tetapi juga tentang bagaimana kelas-kelas tersebut berhubungan satu sama lain. Memahami jenis relasi sangat penting untuk membangun model yang akurat.

Inheritance (Pewarisan)

Merepresentasikan hubungan "is-a" (adalah sebuah). Sebuah kelas turunan (subclass) mewarisi properti dan perilaku dari kelas induk (superclass).

Contoh: Kelas `Anjing` **adalah sebuah** `Hewan`.

Association (Asosiasi)

Hubungan di mana setiap kelas memiliki eksistensi yang independen dan terhubung secara longgar.

Contoh: Seorang `Mahasiswa` mendaftar pada sebuah `MataKuliah`.

Aggregation (Agregasi)

Bentuk asosiasi yang lebih lemah, merepresentasikan hubungan "has-a" (memiliki). Bagian dapat ada secara independen dari keseluruhan.

Contoh: Sebuah `Departemen` **memiliki** beberapa `Karyawan`.

Composition (Komposisi)

Bentuk asosiasi yang kuat di mana bagian tidak dapat ada tanpa keseluruhan. Siklus hidupnya bergantung sepenuhnya.

Contoh: Sebuah `Mobil` **terdiri dari** sebuah `Mesin`.

Activity Diagram

Alur Kerja dan Proses

Activity Diagram memodelkan berbagai **proses** yang terjadi pada sistem.

Diagram ini menggambarkan **runtutan alur kerja** dari satu aktivitas ke aktivitas lainnya secara vertikal, mirip seperti sebuah flowchart.

Sangat baik untuk memvisualisasikan proses bisnis atau logika algoritma yang kompleks.

Sequence Diagram

Interaksi Berdasarkan Waktu

Sequence Diagram menjelaskan interaksi antar objek berdasarkan **urutan waktu**.

Diagram ini dapat menggambarkan urutan atau tahapan pesan (message) yang dikirimkan antar objek untuk menyelesaikan sebuah fungsi atau proses.

Pola Domain Umum dari Tugas Anda

E‑commerce

  • Kelas: User, Produk, Keranjang, Pesanan, Pembayaran
  • Relasi Kunci: User–Pesanan (satu User bisa punya banyak Pesanan)
  • Proses Utama: Browsing → Cart → Checkout → Bayar → Kirim

Streaming

  • Kelas: User, Artist, Lagu, Playlist, Langganan
  • Relasi Kunci: User–Playlist (satu User punya banyak Playlist)
  • Proses Utama: Cari Lagu → Putar → Rekomendasi → Simpan ke Playlist

Media Sosial

  • Kelas: User, Post/Video, Komentar, Like, Notifikasi
  • Relasi Kunci: User–Post (satu User membuat banyak Post)
  • Proses Utama: Buat Konten → Muncul di Feed → Interaksi (Like/Komen)

Transportasi

  • Kelas: Penumpang, Driver, Order, Rute, Pembayaran
  • Relasi Kunci: Penumpang–Order (satu Penumpang bisa punya banyak Order)
  • Proses Utama: Pesan → Cocokkan Driver → Perjalanan → Bayar

Contoh Mini — E‑commerce

Menggabungkan kelas-kelas yang Anda identifikasi dari sistem e-commerce.

Kelas & Atribut

  • User (- id, - nama, - email)
  • Produk (- sku, - nama, - harga, - stok)
  • Pesanan (- noPesanan, - tanggal, - total, - status)
  • ItemPesanan (- qty, - hargaSatuan)
  • Pembayaran (- metode, - referensi)

Relasi & Multiplicity

  • User (1) --- (1..*) Pesanan
  • Pesanan (1) ◆--- (1..*) ItemPesanan (Komposisi)
  • ItemPesanan (1) ---> (1) Produk
  • Pesanan (1) --- (0..1) Pembayaran

Catatan: ItemPesanan tidak dapat ada tanpa Pesanan, oleh karena itu relasinya adalah komposisi (◆).

Aktivitas Kelas (Kelompok)

Langkah (50 menit)

  1. Pilih satu domain: e‑commerce / streaming / media sosial / transportasi / akademik.
  2. (Tugas Utama) Rancang Class Diagram (4-6 kelas) yang merepresentasikan domain. Sertakan atribut, operasi, relasi, dan multiplicity.
  3. (Opsional) Buat Use Case Diagram sederhana (1 Aktor, 3-4 use case) untuk membantu memvalidasi Class Diagram Anda.

Alat yang bisa digunakan: draw.io, StarUML, Lucidchart. Simpan hasilnya sebagai gambar (PNG/SVG).

Keluaran & Presentasi (20 menit)

  • Tampilkan Class Diagram sebagai slide utama.
  • Jelaskan keputusan desain terpenting Anda (pemilihan kelas, relasi, multiplicity).
  • (Opsional) Tunjukkan Use Case Diagram dan jelaskan hubungannya dengan Class Diagram Anda.
Tujuan: Merancang struktur data yang logis dan konsisten melalui Class Diagram.

Rubrik Penilaian

Kelompok (Aktivitas di Kelas)

  • Class Diagram (kelas, atribut, operasi, relasi & multiplicity) — 60%
  • Kejelasan presentasi & argumentasi desain — 25%
  • Use Case Diagram (jika ada, sebagai pendukung) — 5%
  • Kerapian berkas (PNG/SVG, penamaan) — 10%

Individu (Tugas Selanjutnya)

  • Activity Diagram alur utama — 70%
  • Penjelasan singkat proses (≤200 kata) — 20%
  • Kerapian dan format berkas — 10%

Pemeriksaan Kualitas (Hindari Kesalahan Umum)

Jangan Lakukan
  • Menjadikan halaman UI (misal: "Halaman Login") sebagai kelas domain.
  • Menumpuk detail teknis database (seperti foreign key) pada model konseptual.
  • Melupakan multiplicity dan arah relasi antar kelas.
Lakukan Ini
  • Nama kelas adalah kata benda domain, operasi adalah kapabilitas yang bermakna.
  • Gunakan komposisi (◆) saat siklus hidup objek bergantung pada yang lain.
  • Uji konsistensi: pastikan setiap use case dapat didukung oleh kelas dan relasi yang ada.

Tugas Individu (Selanjutnya)

  • Buat Activity Diagram untuk salah satu alur proses utama dari domain yang Anda pilih (contoh: proses checkout, proses upload video, proses pemesanan transportasi).
  • Lengkapi dengan penjelasan singkat (maksimal 200 kata) yang memetakan setiap aksi dalam diagram ke langkah-langkah proses.
  • Ekspor ke format PNG atau SVG. Format nama file: NIM_Nama_Activity.png
  • Waktu pengumpulan akan diinformasikan lebih lanjut.

Terima kasih!

UML adalah alat untuk berpikir dan berkomunikasi: dari kebutuhan → ke struktur → hingga verifikasi alur.
